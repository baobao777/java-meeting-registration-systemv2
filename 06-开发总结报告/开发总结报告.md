## 3. 分层架构

### 后端分层

```
Controller (接收请求/返回响应)
    ↓
Service    (业务逻辑/校验)
    ↓
Mapper     (数据访问/MyBatis)
    ↓
MySQL      (数据持久化)
```

- **Controller 层**：负责参数校验、权限检查、统一响应格式 `ApiResult`
- **Service 层**：核心业务逻辑，如报名时校验会议状态、是否重复、是否满额
- **Mapper 层**：MyBatis 接口 + XML 映射，复杂查询通过 ResultMap 实现关联查

### 前端分层

```
Views (页面组件)
    ↓
Router (路由跳转/导航守卫)
    ↓
API    (Axios 封装)
    ↓
Vite Proxy → Backend API
```

## 4. 核心业务实现

### 4.1 用户注册/登录
- 注册时校验用户名唯一性
- 登录成功写入 HttpSession，前后端基于 Cookie 保持会话
- 前端同时将用户信息写入 localStorage，用于路由守卫判断

### 4.2 会议 CRUD
- 管理员专属操作，控制器层通过 Session 中 `userRole` 校验权限
- 会议列表使用子查询统计当前报名人数 (`registeredCount`)

### 4.3 报名管理
- 报名前校验：(1) 会议状态为 `UPCOMING` (2) 未重复报名 (3) 未满额
- 取消报名为软取消（更新 `status = CANCELLED`），保留记录
- 容量为 0 表示不限制报名人数

### 4.4 路由守卫
- 未登录用户访问受保护页面 → 跳转 `/login`
- 非管理员访问管理后台 → 跳转 `/meetings`

## 5. 数据库设计要点

- 三张表：`user` / `meeting` / `registration`，外键约束保证数据完整性
- `registration` 表设计为软删除（status 字段），保留完整的报名历史
- 会议列表的报名人数通过子查询实时计算，保证数据准确

---

## 6. 异常与边界处理

| 场景 | 处理方式 |
|------|---------|
| 用户名重复 | 返回 400 + 错误提示 |
| 密码错误 | 返回 400 + 错误提示 |
| 重复报名 | 返回 400 + "已报名该会议" |
| 会议满额 | 返回 400 + "会议名额已满" |
| 会议不存在 | 返回 404 + "会议不存在" |
| 未登录操作 | 返回 401 + "未登录" |
| 普通用户越权 | 返回 403 + "无权操作" |
| 前端路由越权 | 路由守卫拦截跳转 |

## 8. 遇到的问题与解决方案

| 问题 | 原因 | 解决 |
|------|------|------|
| MySQL 中文插入报错 "Data too long" | 命令行客户端 charset 设置问题 | 执行 SQL 时加 `--default-character-set=utf8mb4` |
| 进入页面直接看到会议列表 | 路由 `/` 默认跳转到 `/meetings` | 修改为跳转 `/login`，添加路由守卫 |
| 用户登录刷新后状态丢失 | 仅靠内存状态判断登录 | 引入 localStorage 持久化用户信息 |
